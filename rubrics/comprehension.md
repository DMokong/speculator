# Comprehension Rubric

You are evaluating a **comprehension artifact** — a per-AC explanation of how the implementation satisfies the spec, generated by reading the spec and the diff cold (without access to the implementing agent's reasoning). Score each dimension 1-10 using the criteria below.

> **Status:** DRAFT. Calibration examples below are seed examples derived from the Phase 2 design (see `docs/superpowers/specs/2026-04-15-anti-dark-code-pipeline-design.md` in the parent ClaudeClaw workspace). The rubric is not production-ready until each dimension has 10–15 calibrated examples, with explicit "plausible-but-wrong" cases for the Accuracy dimension.

The artifact this rubric scores has two roles, both load-bearing:

1. **As a gate** — catch dark code (implementations that satisfy tests while missing spec intent).
2. **As context** — durable per-AC documentation answering *"why was this written this way?"* for future developers and agents.

**Critical framing:** Assume the implementation is functionally correct (Gate 2 already verified that). Your job is to evaluate whether the *explanation* faithfully and completely describes what the code does *and* whether what the code does matches the spec's intent.

---

## Anti-Inflation Guidance

A score of 7 means *"the explanation is faithful, covers every AC, and matches the spec's intent."* A score of 8 means *"the explanation would let a future engineer modify this code safely without re-reading the diff."* Scores of 9–10 should be rare — reserved for artifacts that name precise file:line locations, surface non-obvious decisions, and explicitly distinguish letter from spirit.

Most first-pass artifacts from a competent agent should score 5–7. Resist the urge to inflate. **A confidently-wrong explanation is worse than no explanation** — false confidence is the failure mode this gate exists to catch.

---

## Dimension 1 — AC Coverage (weight: 0.30)

Does every acceptance criterion have a substantive comprehension entry naming the implementing code?

- **1-3 (Poor):** Multiple ACs have no entry, or entries say things like *"handled in the controller"* without naming files. Entries grouped — one entry covering ACs 1–4 indistinguishably.
- **4-6 (Adequate):** Every AC has an entry, but entries are thin. Code locations are at file granularity but not function/symbol granularity. Some entries are paraphrases of the AC text rather than explanations.
- **7-8 (Good):** Every AC has its own entry. Entries name specific files and functions. Each entry distinguishes the *behavior expected by the AC* from *the mechanism that delivers it.* Partial coverage is explicitly marked.
- **9-10 (Excellent):** Every AC has its own entry with file:line references. Entries explain decisions, not just locations — *"we chose X because Y; the alternative was Z, rejected because W."* Cross-cutting ACs (e.g. an AC that imposes a global constraint) trace through every affected module.

### Calibration Examples

**Score 2-3:**
```yaml
comprehension_entries:
  - ac_id: AC1
    implementation_summary: "Handled in the upload service."
    code_locations: []
    coverage: full
  # AC2, AC3, AC4 absent
```

**Score 5-6:**
```yaml
comprehension_entries:
  - ac_id: AC1
    implementation_summary: >
      Image detection in upload-detector.ts; uploads triggered from there.
    code_locations:
      - file: src/upload-detector.ts
    coverage: full
  - ac_id: AC2
    implementation_summary: "Same module handles oversized files."
    code_locations:
      - file: src/upload-detector.ts
    coverage: full
```
(All ACs present, but entries blur which behavior maps to which AC and locations stay at file granularity.)

**Score 7-8:**
```yaml
comprehension_entries:
  - ac_id: AC1
    ac_text: "Given a PNG file <5MB, when detected in tool output, then upload to active thread"
    implementation_summary: >
      upload-detector.ts:detectImages() scans tool_result blocks for PNG/JPG/GIF
      paths. On match, calls slack-uploader.ts:uploadToThread(channel_id, thread_ts).
    code_locations:
      - file: src/upload-detector.ts
        function: detectImages
      - file: src/slack-uploader.ts
        function: uploadToThread
    coverage: full
```

**Score 9-10:**
```yaml
comprehension_entries:
  - ac_id: AC1
    ac_text: "Given a PNG file <5MB, when detected in tool output, then upload to active thread"
    implementation_summary: >
      upload-detector.ts:detectImages() (lines 34-72) scans tool_result blocks for
      PNG/JPG/GIF paths. Size check at line 51 enforces <5MB before passing to
      slack-uploader.ts:uploadToThread() (lines 18-44), which uses files.uploadV2
      with channel_id and thread_ts threaded through from the caller's context.
      We chose detection-at-tool-result-boundary over filesystem-watch because
      filesystem-watch would race with the model writing partial files.
    code_locations:
      - file: src/upload-detector.ts
        function: detectImages
        lines: "34-72"
      - file: src/slack-uploader.ts
        function: uploadToThread
        lines: "18-44"
    coverage: full
    rejected_alternatives: ["filesystem-watch (race condition)", "post-tool-use hook (latency)"]
```

## Dimension 2 — Accuracy (weight: 0.30)

Does the explanation actually match what the code does? The judge sees both the artifact and the diff — verify, don't trust.

This dimension is the **integrity check.** A confidently-wrong explanation is the dark-code failure mode at scale: the artifact looks reassuring, future readers trust it, and the actual behavior drifts unnoticed. Treat any inaccuracy as a blocking concern unless trivial.

- **1-3 (Poor):** Multiple entries describe behavior the diff does not implement. Function names cited do not exist. Logic flow described does not match the code path.
- **4-6 (Adequate):** Entries are mostly correct at the surface level but include misstatements about edge cases, error handling, or constants ("retries 3 times" when code retries 5; "expires after 24h" when code uses 1h).
- **7-8 (Good):** Every entry's claims are verifiable in the diff. Edge cases described match edge cases handled. Constants in the explanation match constants in the code.
- **9-10 (Excellent):** Entries are precise enough that a reviewer could spot-check claims by reading file:line references and find them confirmed. Where the spec is ambiguous, the explanation explicitly states which interpretation the code chose and why.

### Calibration Examples

**Score 2-3:**
```yaml
- ac_id: AC3
  ac_text: "Token rotation every 24 hours"
  implementation_summary: >
    Tokens are rotated every 24h by a background job that runs at midnight UTC.
    On rotation, old tokens are revoked and dependents notified via webhook.
```
*Diff shows*: tokens rotate every 1h, no webhook code, no background job — all three claims are wrong. Looks plausible to a quick reader; demolishes trust if anyone checks.

**Score 5-6:**
```yaml
- ac_id: AC2
  ac_text: "Failed uploads retry with exponential backoff"
  implementation_summary: >
    Retries up to 3 times with 1s, 2s, 4s backoff.
```
*Diff shows*: retries 5 times with 1s, 2s, 4s, 8s, 16s. The shape of the explanation is right; the constants are wrong. Future reader assumes 3 retries, designs around it, gets surprised.

**Score 7-8:**
```yaml
- ac_id: AC2
  ac_text: "Failed uploads retry with exponential backoff"
  implementation_summary: >
    upload-handler.ts:retryUpload() retries up to 5 times with exponential backoff
    starting at 1s and doubling (1s, 2s, 4s, 8s, 16s). The cap is set by
    MAX_RETRY_ATTEMPTS = 5 in config/upload.ts.
```
*Diff confirms*: retry count, backoff schedule, and constant name all match.

**Score 9-10:**
```yaml
- ac_id: AC2
  ac_text: "Failed uploads retry with exponential backoff"
  implementation_summary: >
    upload-handler.ts:retryUpload() (lines 88-114) retries up to MAX_RETRY_ATTEMPTS
    (=5, config/upload.ts:12) with exponential backoff starting at INITIAL_BACKOFF_MS
    (=1000) and doubling. Spec says "exponential backoff" without specifying base or
    cap; we chose 1s/×2/5-attempts to land worst-case retry at 16s, comfortably
    inside the 30s pipeline timeout. Failures past the 5th attempt surface to the
    caller as RetriesExhaustedError, defined in errors.ts:42.
```

## Dimension 3 — Intent Alignment (weight: 0.25)

Does the implementation match the *spirit* of the spec, not just the letter? **This is the dark-code detector.** A correct, faithful explanation can still score low here if what the explanation describes does not deliver the user-visible outcome the spec intended.

The canonical example: spec says *"Users can delete their account."* Implementation does soft-delete (sets `deleted_at`, hides from queries). Tests pass because they check the account is no longer reachable through the API. Explanation accurately says *"DELETE /account sets deleted_at and removes the row from active queries."* The explanation is precise. The test passes. **The user's data is still there.** That's letter without spirit. Score it low.

- **1-3 (Poor):** Multiple entries describe behavior that satisfies the AC text but defeats the spec's stated purpose (problem statement, anti-patterns, user journeys). The implementation games the AC.
- **4-6 (Adequate):** Most entries align with intent, but at least one AC's implementation chose a defensible-but-narrower interpretation of intent without flagging it.
- **7-8 (Good):** Every entry's described behavior matches what the spec was actually trying to achieve. Where the spec is ambiguous, the entry calls out the interpretation and the alternative.
- **9-10 (Excellent):** Entries explicitly name which spec anti-patterns the implementation avoids and how. Where intent could be gamed (the AC alone wouldn't catch a violation), the entry shows the additional checks that protect against it.

### Calibration Examples

**Score 2-3:**
```yaml
spec_problem_statement: "Users complain that 'delete my account' leaves their data
   accessible to admins and to ML training pipelines. We need actual erasure."
spec_anti_patterns: ["Soft-delete is NOT acceptance of the AC."]
- ac_id: AC1
  ac_text: "DELETE /account succeeds and the account is no longer reachable"
  implementation_summary: >
    DELETE /account sets accounts.deleted_at = NOW() and adds a deleted_at IS NULL
    filter to all read queries. Account no longer appears in lists.
  coverage: full
```
*The explanation is accurate; the implementation satisfies the literal AC; the spec's actual intent (erasure) is violated. Anti-pattern from the spec is unaddressed.*

**Score 5-6:**
```yaml
- ac_id: AC1
  ac_text: "Webhook retries on transient failures"
  implementation_summary: >
    Retries on HTTP 5xx via exponential backoff. 4xx responses are not retried.
```
*Spec problem statement said "downstream services need eventual delivery guarantees" but didn't enumerate which status codes count as transient. Choosing not to retry 429 (rate-limited, transient) is defensible but narrows the spec's intent without flagging it.*

**Score 7-8:**
```yaml
- ac_id: AC1
  ac_text: "Account deletion removes user data"
  implementation_summary: >
    DELETE /account triggers DataPurgeJob (data-purge.ts:17), which removes the
    accounts row, all foreign-key referenced rows in user_data, and queues an S3
    object-deletion job for any user-uploaded files. Soft-delete was rejected
    (see spec anti-patterns). The job emits a deletion-complete event consumed
    by the ML training pipeline to drop training data on read.
  coverage: full
  intent_notes: >
    Spec's intent ("actual erasure, not hide") is enforced by the data-purge job
    and the ML pipeline event — both required to deliver the spec's outcome.
```

**Score 9-10:**
```yaml
- ac_id: AC1
  ac_text: "Account deletion removes user data"
  implementation_summary: >
    [as above, plus:]
    Anti-pattern protection: DataPurgeJob runs synchronously through the API
    response (data-purge.ts:42 awaits all deletes), so a "successful" DELETE
    response cannot precede actual erasure. The 30s timeout is enforced; if
    purge fails, the API returns 500 and the account is NOT marked deleted —
    avoiding the "looks deleted, isn't" failure mode named in spec.anti_patterns.
  rejected_alternatives:
    - "Async job (rejected: violates spec intent — user gets success before erasure)"
    - "Soft-delete + nightly purge (rejected: same)"
```

## Dimension 4 — Scope Containment (weight: 0.15)

Are there behaviors in the diff that no AC asked for? Unexplained behaviors are either scope creep, dark-code pockets, or load-bearing utility additions the spec should have enumerated.

This dimension is about *positive listing of what exists* — the artifact's `unexplained_behaviors` section. The judge's job is to evaluate whether each unexplained behavior is appropriately classified.

- **1-3 (Poor):** The diff contains substantive new behaviors that no AC covers, and they are absent from the unexplained_behaviors section. The artifact silently endorses scope creep.
- **4-6 (Adequate):** Major unexplained behaviors are listed, but classification is inconsistent (a feature addition labelled `minor_utility`; a renamed helper labelled `scope_creep`).
- **7-8 (Good):** Every behavior in the diff that is not directly required by an AC is listed and correctly classified as `minor_utility` (small helper, refactor, no user-visible behavior change) or `scope_creep` (new user-visible behavior beyond spec).
- **9-10 (Excellent):** Unexplained behaviors include a recommendation: promote to a new AC, defer to a future spec, document in spec out-of-scope, or remove from the diff. The artifact treats scope-creep findings as actionable.

### Calibration Examples

**Score 2-3:**
```yaml
unexplained_behaviors: []
```
*Diff contains:* a new `.webp` extension check, a refactored shared utility module, and a new admin-only endpoint for force-resync — none in any AC, none listed.

**Score 5-6:**
```yaml
unexplained_behaviors:
  - description: "Refactored shared util module"
    file: src/util/shared.ts
    concern: scope_creep    # mis-classified — refactor with no behavior change is minor_utility
  - description: ".webp extension support"
    file: src/upload-detector.ts
    concern: minor_utility   # mis-classified — new user-visible behavior is scope_creep
```

**Score 7-8:**
```yaml
unexplained_behaviors:
  - description: "Refactored shared util module to extract handleResult helper"
    file: src/util/shared.ts
    line_range: "12-44"
    concern: minor_utility
  - description: ".webp extension support added to image detection"
    file: src/upload-detector.ts
    line_range: "47-49"
    concern: scope_creep
```

**Score 9-10:**
```yaml
unexplained_behaviors:
  - description: ".webp extension support added to image detection"
    file: src/upload-detector.ts
    line_range: "47-49"
    concern: scope_creep
    recommendation: >
      Either promote to a new AC ("AC8: PNG/JPG/GIF/WEBP supported") or move to
      spec out-of-scope. Today this is functional but undocumented — a future
      spec that enumerates "supported formats" would make a contradictory claim.
  - description: "Refactored handleResult into shared util"
    file: src/util/shared.ts
    line_range: "12-44"
    concern: minor_utility
    recommendation: "OK to ship; no spec impact."
```

## Scoring

### Default Weights

| Dimension          | Weight |
|-------------------|--------|
| AC Coverage        | 0.30   |
| Accuracy           | 0.30   |
| Intent Alignment   | 0.25   |
| Scope Containment  | 0.15   |

Round the overall score to one decimal place.

### Per-Dimension Minimum

Each dimension must score >= 5. Any dimension below 5 forces a `fail` regardless of the weighted overall.

The per-dimension minimum defaults to 5 but can be overridden in `.claude/sdlc.local.md` via `gates.comprehension.per_dimension_minimum`.

> **Why higher than Gate 2b's minimum of 4?** Gate 2c is the dark-code-prevention gate. A weak dimension here means future readers cannot trust the artifact — the failure mode isn't a noisy gate, it's silent comprehension drift. Higher minimums protect the artifact's value as durable context.

---

## Flags

### Blocking
Issues that must be fixed before the gate can pass, even if the overall score meets the threshold. Any blocking flag forces a `fail` result.

- Any `Accuracy` finding where the explanation demonstrably contradicts the diff (function does not exist; constant value misstated; logic path described does not exist in code)
- Any AC marked `coverage: missing` without an accompanying override-justification entry in the spec
- Any unexplained behavior classified `scope_creep` for a user-visible feature addition
- An `Intent Alignment` finding where the spec's anti-patterns section names the chosen implementation as forbidden

### Recommended
Issues that should be addressed but don't block.

- AC entries that name only files (no functions, no line ranges) when the spec is non-trivial
- `Intent Alignment` entries that don't surface rejected alternatives where the spec was ambiguous
- `Scope Containment` entries lacking a recommendation field
- Any single dimension scoring below 6

### Advisory
Nice-to-have improvements at the author's discretion.

- Opportunities to expand `intent_notes` with more specific anti-pattern protection
- Cross-references between related ACs that share a single implementation site
- Suggestions to promote `minor_utility` items into the spec out-of-scope section for clarity

---

## Gate Decision

The gate passes only when **all three conditions** are met:

1. **Overall score >= threshold** (from `gates.comprehension.threshold` in `sdlc.local.md`, default **7.0**)
2. **Every dimension >= per-dimension minimum** (default **5**)
3. **No blocking flags**

If any condition fails: `result: fail`
If all pass: `result: pass`

When failing, clearly state which condition(s) caused the failure so the artifact author (the comprehension agent on a re-run, or a human revising) knows exactly what to fix.

---

## Calibration Set Requirements

Before this rubric is production-ready, build a calibration set with at least:

- **10–15 examples per dimension** at each scoring band (1-3, 4-6, 7-8, 9-10).
- **At least 5 "plausible-but-wrong" examples for Accuracy.** This is the highest-value class — explanations that read confidently but contradict the diff. Without them, the LLM judge will tend toward charitable scoring.
- **At least 3 "letter-vs-spirit" examples for Intent Alignment.** Soft-delete-vs-erasure is the canonical seed; add domain-varied cases (e.g. session expiry, rate limiting, retry semantics).
- **At least 3 scope-creep classification edge cases** for Scope Containment (a refactored helper, a feature that's in the diff but wasn't asked for, a renamed function with different semantics).

Track judge-vs-human divergence using the same `spec-bench calibrate` flow used for Gate 1 and Gate 2b. A divergence > 1 point on any dimension flags the rubric as `needs_tuning`.
