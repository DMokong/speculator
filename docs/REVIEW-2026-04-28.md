## TL;DR

Speculator is in a healthy place architecturally — the 4-gate (now 5-gate, with Gates 2a + 2b) pipeline is internally consistent, the rubrics are well-calibrated with anti-inflation guidance, and the dogfooding evidence (SPEC-001 with all four gate artifacts) shows the pipeline can ship its own changes end-to-end. The Spec-Bench harness is real, runnable, and 67/67 tests green. The thesis ("quality in = quality out") is now backed by an empirical apparatus, not just narrative.

The weak spots are documentation drift and surface-area risk, not architectural debt:

1. `plugin.json` says `2.7.0` but the CHANGELOG, git log, and shipped behaviour are at `2.8.0`. **Bug.**
2. The README's Plugin Structure tree is missing 1 skill (`eval-authoring`), 2 agents (`eval-intent-scorer`, `eval-quality-scorer`), and 2 rubrics (`eval-intent.md`, `eval-quality.md`). Roadmap aspirations have shipped without showing up in the inventory.
3. The Gate Pipeline diagram in README still shows the original 4 gates only — Gates 2a (eval-intent) and 2b (eval-quality) are explained later in prose but absent from the "headline" diagram.
4. Spec-Bench is invisible from the main README. A benchmark with 67 tests, three-round results, and a calibration protocol is undersold.
5. The MANIFESTO's "Roadmap" still lists "spec quality benchmark dataset" as future work even though `benchmarks/results/3-round-spec-quality.yml` exists and has been published.

These are all fixable in one doc-update pass. Verdict: **ship the next version with the docs caught up; don't touch the architecture.**

## Strengths

### 1. Architecture is coherent, layered, and dogfooded

The Skills-as-orchestration / Agents-as-workers / Rubrics-as-instructions split is doing its job. Skills handle conversation flow and tool use; agents are stateless evaluators that produce YAML evidence; rubrics are the "constitution" the agents read. This is good plugin hygiene — each piece has a single concern and can be swapped (you could replace `spec-scorer` with a different model and the rest still works).

The dogfooding evidence is concrete: `docs/specs/eval-quality-scoring/evidence/` contains all four gate artifacts (`gate-1-scorecard.yml` 8.0/10 PASS, `gate-2-quality.yml`, `gate-3-review.yml`, `gate-4-summary.yml`). SPEC-001 was shipped through Speculator's own pipeline. This is the cleanest possible answer to "does the thing actually work?"

### 2. Rubrics are calibrated, not just rubric-shaped

`rubrics/spec-quality.md` does what most LLM-as-judge rubrics fail to do: it bands the scoring (1-3, 4-6, 7-8, 9-10) and shows worked calibration examples per band per dimension. The "anti-inflation guidance" up top ("a score of 7 means ready for implementation; 9-10 should be rare") is exactly the kind of language that fights LLM grade inflation. Same pattern repeats in `eval-intent.md` and `eval-quality.md`. This is the highest-craft work in the repo.

### 3. The self-improvement loop has an explicit boundary constraint

`skills/sdlc-run/SKILL.md` Section 6 spells out what the loop may and may not do: it can add detail, examples, anti-patterns, and clarifications, but it MUST NOT remove or alter requirements/ACs/scope/risk-level without human approval. This boundary is repeated in three places (the philosophy preamble, Phase 1, Section 6) — appropriately defensive for a feature that, if mis-scoped, would let an autopilot quietly rewrite a spec until it scored well. Designing around the failure mode rather than designing it away.

### 4. Worktree isolation + spec resolution is properly engineered

`lib/spec-resolution.md` is short but covers the resolution order (explicit > worktree affinity > lock check > single-spec fallback > ask user), lock semantics (with stale-lock detection), and the cross-worktree redirect check that every skill calls *before* operating. This catches the most likely production-mode footgun: running `/sdlc score` on `main` when the spec lives in a worktree.

### 5. Mandatory secrets scan in Gate 3 is the right kind of paranoia

`agents/code-reviewer/AGENT.md` doesn't trust the reviewer to *read* code carefully — it makes them grep, in five categories, with a "when in doubt, flag it" principle and an auto-fail rule for any true positive. Then `tests/test-secrets-scan.sh` validates the patterns against fixtures (17 true-positive checks, 8 false-positive checks, all 25 PASS). The skill description eval added in v2.6.0 is the same pattern applied to a different failure mode (skills with under-triggering descriptions). Gate 3 treats security and skill quality as auditable rather than vibes-based.

### 6. Spec-Bench is the missing measurement, partially built

`benchmarks/` is a real evaluation harness, not a marketing demo. It runs an 8-target matrix (3 LLMs × 2 processes × 2 harnesses), produces a 3-round spec-quality dashboard, has 67 passing unit tests for the orchestrator/scoring/review/calibration pipeline, and includes a human-in-the-loop calibration protocol that flags judge-vs-human divergence > 1 point per dimension as `needs_tuning`. That last piece is the part that elevates this from "we ran specs through scoring" to "we have an apparatus for trusting the scores." This is the closest thing in the wild to a SWE-bench equivalent for spec quality, and it's in this repo. It deserves a top-line README mention.

## Concerns

### C1. **[Bug]** `plugin.json` version is stale

```json
"version": "2.7.0"
```

CHANGELOG.md tops out at `2.8.0 — Eval-First SDLC: Pre-Implementation Eval Authoring (SPEC-040)`, and the most recent commit (`6703adc`) is "chore: bump version to 2.8.0". The manifest didn't get the memo. **Fix: bump to `2.8.0`.**

### C2. **[Doc drift]** README inventory missing v2.7.0 + v2.8.0 additions

The Plugin Structure tree (README lines 396–428) is what users will scan to understand "what's in this thing." Today it says:

- 8 skills (missing `eval-authoring`)
- 3 agents (missing `eval-intent-scorer` and `eval-quality-scorer`)
- 6 rubrics (missing `eval-intent.md` and `eval-quality.md`)

A user who installs the plugin today gets 9 skills, 5 agents, and 8 rubrics — a third of the surface area is not in the inventory. The CHANGELOG records all of these additions; the README never picked them up.

### C3. **[Doc drift]** Gate Pipeline diagram is the old one

README line 217:

```
Gate 1: Spec Quality --> Gate 2: Code Quality --> Gate 3: Review --> Gate 4: Evidence Package
```

`skills/sdlc-run/SKILL.md` Pipeline Position Detection table actually lists 6 phases (Phase 1 / Phase 2 / Phase 2a / Phase 3 / Phase 3a / Phase 4 / Phase 5). The "headline" diagram in the README hides the two opt-in gates that are the entire story of the last two minor versions. They are explained 100 lines later in the body, but a reader who only skims will miss them.

### C4. **[Doc drift]** Spec-Bench is invisible from the top-level README

The MANIFESTO's "Call" section (Section VII) literally argues "we need a spec quality benchmark... we couldn't find one... we're starting." Spec-Bench is the answer. It is in this repo. It is not linked from the top-level README. A reader following the MANIFESTO's call will not find that the call has been partially answered without spelunking into `benchmarks/`.

### C5. **[Doc drift]** MANIFESTO roadmap is no longer accurate

The Roadmap (Section VII) lists "Spec quality benchmark dataset — the SWE-bench for specifications, community-contributed and openly scored" as future work. The harness ships; there is a 3-round spec-quality result set in `benchmarks/results/`. The framing should shift to *"Spec-Bench is built and producing results; the next step is community contribution and a public leaderboard."*

### C6. `/sdlc implement` is documented but doesn't have its own skill file

Every other subcommand (`start`, `score`, `gate`, `status`, `doctor`, `compact`, `run`, `eval`) routes to a dedicated `skills/<name>/SKILL.md`. `implement` is described inline in `skills/sdlc/SKILL.md` lines 47–98. The structure is fine for now, but it's the one entry that breaks the symmetry, and there's nothing wrong with `implement` becoming its own skill the next time you touch it. (Not gate-blocking. Just a tidiness note.)

### C7. Calibration burden is rising

Counting just the configurable thresholds in `sdlc.local.md`:
- `gates.spec-quality.threshold` (7.0)
- `scoring.dimension_minimum` (5)
- `scoring.weights` (6 weights summing to 1.0)
- `gates.code-quality.coverage_threshold` (80) and four boolean checks
- `gates.eval-intent.threshold` (6.5), `per_dimension_minimum` (4), `max_eval_retries` (3)
- `gates.eval-quality.threshold` (6.5), `per_dimension_minimum` (4) — implicit
- `run.self_improvement_trigger` (8.0), `full_auto_threshold` (7.8), `guided_threshold` (7.0)
- `run.intent_verifiability_min` (8) — hard gate
- `run.max_spec_retries` (3), `max_code_retries` (3)
- `run.risk_signals` (keyword list)

This is a lot of dials to know how to set. The `/sdlc doctor --init` template gives reasonable defaults, but the project doesn't (yet) have guidance on *when to retune*. Suggestion (post-v3): a `docs/calibration-guide.md` showing the empirical pass/fail distribution from Spec-Bench and proposing default tunings per project profile (greenfield, refactor, brownfield).

### C8. Gate 2 has a soft underbelly for "non-code" specs

`rubrics/code-quality.md` is evidence-based: tests pass, coverage hits threshold, optionally lint/typecheck/build. For a spec that only changes prompts/skills/agents (Speculator's own SPEC-040 was largely this), there are no Python or TS tests to run, and the gate collapses to "the shell tests still pass." This is pragmatic and the CHANGELOG even calls it out (v2.7.0 SPEC-001 "all-markdown changeset, 25/25 tests pass") — but the loophole isn't explicit anywhere in the rubric. A paragraph in `code-quality.md` saying *"For prompt-only changes, Gate 2 is satisfied by the structural test suite + a documented N/A rationale; do not invent fragile LLM tests"* would close the gap and match how the project actually behaves. (Aligns with claudeclaw's own preference: "Gate 2 for prompt-only changes: record as N/A with rationale, not fragile LLM tests.")

### C9. Risk validation uses LLM judgment for false-positive filtering

`agents/spec-scorer/AGENT.md` Risk Validation step 4: when risk signals are matched and `risk_level` is `low`, the agent uses "LLM judgment to filter false positives (e.g., 'delete a todo item' is not a dangerous deletion)." This is correct — keyword matching on "delete" without context would be wrong almost every time — but it's a non-deterministic call by the same model that just scored the spec. The risk signals list is in config, but the *filter* is invisible. This is fine for v2.x but is a place where calibration data (when does the filter agree with humans?) would matter eventually.

### C10. No CI / no automated release pipeline visible in the repo

Tests are runnable but there's no `.github/workflows/*.yml`, no `Makefile`, no `pre-commit`. The release process (per claudeclaw's memory: plugin repo → marketplace repo → local cache copy, with `.claude/` paths sometimes blocking even Bash mkdir) lives entirely in tribal knowledge. Adding even a minimal `make test` target and a one-page `RELEASE.md` would let contributors land a PR without needing to know which steps are landmines.

### C11. AGENTS.md is mostly redundant

`AGENTS.md` (40 lines) duplicates the "landing the plane" guidance that lives in CLAUDE.md across most projects, plus a short bd cheatsheet. It could either become a real "agents in this repo do X" guide (since Speculator literally orchestrates agents) or be deleted in favour of the README. Currently it's neither.

### C12. SYSTEM-SPEC.md has only one entry — the living-spec thesis is undertested

35 lines, one spec compacted (SPEC-001). The MANIFESTO Section VI ("Specs as Living Organisms") is one of the strongest pieces of writing in the repo, and its premise depends on having *many* shipped specs folded in over time so the provenance trail and amend mechanics can demonstrate value. Today it's a single domain with five sub-sections from one spec. This is not a defect — it's just early. Worth noting because the next 2-3 SPECs that close should be pushed through `/sdlc compact` aggressively to build evidence; the agent has a `<500-line` sizing constraint hardcoded as a known cliff, but we're nowhere near it.
